# Data Flow

## Configuration to Execution

```
WORKFLOW.md (target repo)
       │
  ┌────▼────┐
  │ Workflow │  Parse YAML front matter
  │ Loader   │  + Liquid prompt body
  └────┬────┘
       │
  ┌────▼────┐
  │ Config   │  Typed ServiceConfig with
  │ Layer    │  env-var ($VAR) resolution
  └────┬────┘
       │
  ┌────▼──────────────────────┐
  │       Orchestrator        │
  │  poll → reconcile →      │
  │  dispatch → retry        │
  └──┬────────┬────────┬────┘
     │        │        │
┌────▼───┐ ┌──▼────┐ ┌─▼──────────┐
│Tracker │ │Work-  │ │ Agent      │
│Client  │ │space  │ │ Runner     │──▶ DB
│(GitHub/│ │Manager│ │(claude-    │   (libsql)
│ Asana) │ │       │ │ agent-sdk) │
└────────┘ └───────┘ └────────────┘
```

## Orchestrator Tick Cycle

Each poll tick executes in order:

1. **Reconcile** — Refresh running issue states from the tracker; terminate workers for terminal/non-active issues; detect stalled agents
2. **Validate** — Re-check config validity (supports live reload via file watcher)
3. **Create tracker adapter** — Instantiate from config. On adapter error, log and skip remaining steps
4. **Process watched states** — Dispatch agents for issues in watched states (e.g., "Human Review") that have review activity
5. **Fetch candidates** — Poll active issues with optional assignee/label filters
6. **Sort and dispatch** — Priority ascending, then oldest first. Check concurrency limits. Create workspace, run hooks, start agent session
7. **Schedule next tick** — Wait `polling.interval_ms` before repeating

## Agent Session Lifecycle

1. **`createWorkspace()`** — Creates or reuses a per-issue directory (or git worktree). Runs `after_create` hook on first creation
2. **`runBeforeRunHook()`** — Executes the optional `before_run` shell hook
3. **`startSession()`** — Validates workspace path and assigns a local session UUID
4. **`runTurn()`** — Calls `query()` from `@anthropic-ai/claude-agent-sdk` with the rendered prompt. Translates SDK messages into orchestrator events
5. **Multi-turn** — After each turn, refreshes issue state; continues if still active and under `max_turns`
6. **`runAfterRunHook()`** — Executes the optional `after_run` shell hook
7. **On exit** — Normal exits schedule 1s continuation retry; failures schedule exponential backoff

## Error Handling

The codebase uses discriminated union types for expected errors:

| Error Type | Source |
|-----------|--------|
| `TrackerError` | Network failures, API errors from GitHub/Asana |
| `ValidationError` | Missing or invalid config fields |
| `WorkflowError` | YAML parse errors, missing files |
| `PromptBuildError` | Liquid template parse/render failures |
| `InitError` | GitHub API failures during init |

Each error type has a `code` field for programmatic matching and corresponding type guards (`isTrackerError()`, etc.).
